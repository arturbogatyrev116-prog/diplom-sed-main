3.2. Описание базы данных

3.2.1. Общая характеристика

В качестве СУБД используется SQLite — встраиваемая реляционная база данных, не требующая отдельного серверного процесса. Взаимодействие с базой данных осуществляется через ORM-фреймворк Prisma, который обеспечивает типобезопасный доступ к данным, автоматическую генерацию клиента и управление миграциями схемы.

База данных содержит 9 моделей (таблиц), охватывающих все аспекты функционирования системы: управление пользователями, жизненный цикл документов, маршруты согласования, электронные подписи, вложения, комментарии и журнал аудита.

3.2.2. Перечисляемые типы

Схема определяет четыре перечисляемых типа, обеспечивающих целостность данных на уровне базы:

- UserRole — роль пользователя: EMPLOYEE, MANAGER, OWNER, ADMIN;
- DocumentStatus — статус документа: DRAFT, ON_APPROVAL, REVISION_REQUIRED, APPROVED, REJECTED, SIGNED, ARCHIVED;
- DocumentType — тип документа: CONTRACT, ORDER, STATEMENT, INVOICE, MEMO;
- ApprovalStepStatus — статус шага согласования: PENDING, APPROVED, REJECTED, RETURNED_FOR_REVISION.

3.2.3. Описание моделей

Модель User (Пользователь) хранит учётные данные и профиль каждого пользователя системы. Поле passwordHash содержит хэш пароля, вычисленный алгоритмом bcrypt. Поля mfaSecret и mfaVerifiedAt используются для хранения секрета TOTP и момента активации многофакторной аутентификации соответственно. Флаг isActive позволяет деактивировать учётную запись без её удаления.

Модель Document (Документ) является центральной моделью системы. Хранит содержимое документа, его текущий статус и тип. Поле contentHash содержит SHA-256-хэш содержимого документа на момент подписания — используется при верификации ЭЦП. Поле currentVersion отражает номер актуальной версии. Поле deletedAt реализует паттерн мягкого удаления (soft delete): документ не удаляется физически, а помечается временной меткой удаления. Поле archivedAt фиксирует момент архивации. Индексы созданы по полям status, type, authorId, createdAt и deletedAt для ускорения типичных запросов фильтрации и сортировки.

Модель DocumentVersion (Версия документа) реализует версионирование: при каждом сохранении изменений создаётся новая запись со снимком заголовка (titleSnapshot) и содержимого (contentSnapshot). Уникальное ограничение (documentId, versionNumber) исключает дублирование версий. При удалении родительского документа все версии удаляются каскадно.

Модель Comment (Комментарий) хранит комментарии пользователей к документам. Связана с документом и автором. Каскадное удаление обеспечивает автоматическое удаление комментариев при удалении документа.

Модель ApprovalRoute (Маршрут согласования) описывает маршрут согласования конкретного документа. Связь с документом — один к одному. Содержит набор шагов согласования (ApprovalStep). Удаляется каскадно вместе с документом.

Модель ApprovalStep (Шаг согласования) описывает один шаг в маршруте согласования: назначенного согласующего (approverId), порядковый номер шага (stepOrder), текущий статус и необязательный комментарий к решению. Уникальное ограничение (routeId, stepOrder) исключает дублирование шагов в одном маршруте.

Модель Attachment (Вложение) хранит метаданные загруженных файлов-вложений: оригинальное имя файла, путь к файлу в файловой системе, MIME-тип и размер в байтах. Физические файлы хранятся в директории data/attachments/{documentId}/. Каскадное удаление обеспечивает согласованность метаданных при удалении документа.

Модель AuditLog (Журнал аудита) хранит записи о всех значимых действиях в системе. Поле actorId допускает значение NULL для фиксации системных событий. Поле detailsJson содержит дополнительные сведения о событии в формате JSON. Фиксируются IP-адрес и User-Agent инициатора действия. Индексы по actorId, (entityType, entityId) и createdAt обеспечивают эффективную фильтрацию журнала. При удалении пользователя поле actorId обнуляется, что сохраняет историю событий.

Модель Signature (Электронная подпись) хранит метаданные ЭЦП документа. Связь с документом — один к одному. Поле contentHash фиксирует хэш содержимого на момент подписания. Поле signaturePath указывает путь к файлу подписи в формате PKCS#7 (.p7s) в директории data/signatures/. Операция удаления документа ограничена — нельзя удалить подписанный документ без предварительного удаления подписи.

3.2.4. Каскадные операции и ссылочная целостность

В схеме применяются три стратегии обработки связанных записей при удалении родительской сущности:

- Cascade — дочерние записи удаляются автоматически. Применяется для версий, комментариев, вложений, маршрутов и шагов согласования: удаление документа влечёт полную очистку связанных данных;
- Restrict — удаление запрещено при наличии связанных записей. Применяется для подписей и авторов документов: нельзя удалить пользователя, являющегося автором документов, или документ, имеющий подпись;
- SetNull — ссылка обнуляется. Применяется в журнале аудита: при удалении пользователя его записи в журнале сохраняются, но поле actorId обнуляется.

Такое сочетание стратегий обеспечивает целостность данных и одновременно сохраняет историческую информацию даже после удаления учётных записей.

3.2.5. Soft delete и версионирование

Паттерн мягкого удаления реализован через поле deletedAt в модели Document. Документ считается удалённым, если deletedAt не равно NULL. Все политики доступа и запросы к документам явно проверяют это условие, исключая «удалённые» документы из выборок. Физическое удаление записи не производится, что позволяет при необходимости восстановить документ или провести аудит.

Версионирование документов реализовано через модель DocumentVersion: при каждом сохранении изменений создаётся снимок текущего состояния заголовка и содержимого. Номер версии хранится непосредственно в модели Document (поле currentVersion) и инкрементируется при каждом обновлении.
