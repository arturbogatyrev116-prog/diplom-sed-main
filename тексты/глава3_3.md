3.3. Средства аутентификации и управления доступом

3.3.1. Реализация входа в систему

Аутентификация пользователей реализована на основе библиотеки NextAuth 5 с использованием провайдера Credentials — механизма проверки подлинности по паре «электронная почта / пароль». Сессия пользователя управляется через JWT-токены (стратегия jwt), что исключает необходимость хранения сессий на сервере.

Процедура аутентификации выполняется в следующем порядке:

- пользователь вводит адрес электронной почты и пароль на странице входа (/login);
- сервер извлекает запись пользователя из базы данных по адресу электронной почты (email приводится к нижнему регистру);
- проверяется флаг активности учётной записи (isActive); деактивированные учётные записи отклоняются без проверки пароля;
- введённый пароль сравнивается с хранимым хэшем с помощью функции bcrypt.compare() библиотеки bcryptjs; пароли в базе данных хранятся исключительно в виде bcrypt-хэшей;
- если у пользователя включена многофакторная аутентификация (mfaEnabled = true), дополнительно проверяется одноразовый TOTP-код;
- при успешной аутентификации в журнал аудита записывается событие LOGIN_SUCCESS с указанием метода входа, IP-адреса и User-Agent клиента;
- данные пользователя (id, email, fullName, role, isActive) помещаются в JWT-токен и становятся доступны через объект сессии на всех страницах приложения.

Маршрут /api/auth/callback/credentials защищён ограничением частоты запросов: не более 20 POST-запросов за 10 минут с одного IP-адреса. При превышении лимита возвращается ответ 429 Too Many Requests с заголовком Retry-After.

3.3.2. Многофакторная аутентификация (TOTP)

Система поддерживает опциональную многофакторную аутентификацию на основе алгоритма TOTP (Time-based One-Time Password, RFC 6238), реализованного библиотекой otplib.

Настройка МФА выполняется пользователем в разделе настроек:

- клиент отправляет POST-запрос на /api/mfa/setup; сервер генерирует случайный base32-секрет и формирует URI в формате otpauth:// с параметрами: эмитент — «Diplom SED», метка — адрес электронной почты пользователя;
- URI передаётся клиенту и отображается в виде QR-кода, который пользователь сканирует приложением-аутентификатором (Google Authenticator, Яндекс Ключ и др.);
- для подтверждения настройки пользователь вводит первый сгенерированный код и отправляет его на /api/mfa/confirm; сервер проверяет код функцией verify({ secret, token }); при успехе в базе данных устанавливаются флаг mfaEnabled = true, значение mfaSecret и временная метка mfaVerifiedAt.

Проверка МФА при входе: если у пользователя включена МФА, поле totpCode в форме входа обязательно. При отсутствии кода выбрасывается исключение MFARequiredError — клиент получает сигнал отобразить поле ввода кода. Код нормализуется: удаляются все нецифровые символы, длина ограничивается шестью знаками. Код проверяется функцией verify() библиотеки otplib; при несовпадении аутентификация отклоняется.

Эндпоинты /api/mfa/setup и /api/mfa/confirm также защищены rate limiting с теми же параметрами, что и эндпоинт входа.

3.3.3. Ролевая модель управления доступом (ABAC)

Управление доступом реализовано по модели ABAC (Attribute-Based Access Control) — права определяются не только ролью пользователя, но и атрибутами конкретного объекта доступа (статус документа, авторство, признак удаления).

Система определяет четыре роли:

- EMPLOYEE — рядовой сотрудник, создаёт и редактирует собственные документы, отправляет их на согласование;
- MANAGER — менеджер, обладает теми же правами, что и EMPLOYEE, плюс право принимать решения по согласованию документов;
- OWNER — руководитель организации, подписывает утверждённые документы ЭЦП и управляет архивом;
- ADMIN — системный администратор, имеет доступ только для чтения к завершённым документам, управляет пользователями и просматривает журнал аудита.

Политики доступа реализованы в виде чистых функций в модуле server/policies/document.ts. Каждая функция принимает субъект доступа (DocumentSubject: userId + role) и объект доступа (DocumentPolicyDoc: authorId + status + deletedAt) и возвращает булево значение. Все политики явно проверяют признак мягкого удаления (deletedAt == null) — удалённые документы недоступны ни одной роли.

Ключевые политики:

- canCreateDocument — разрешено EMPLOYEE и MANAGER; запрещено OWNER и ADMIN;
- canViewDocument — разрешено только автору документа (любая роль, кроме ADMIN и OWNER);
- canViewDocumentForSigning — разрешено OWNER для документов со статусом APPROVED, SIGNED или REJECTED;
- canViewDocumentAsAdminReadOnly — разрешено ADMIN для завершённых документов (APPROVED, REJECTED, ARCHIVED, SIGNED);
- canEditDocument / canUpdateDocument — разрешено автору документа при статусе DRAFT или REVISION_REQUIRED; запрещено ADMIN;
- canSubmitForApproval — разрешено автору документа (EMPLOYEE или MANAGER) при статусе DRAFT или REVISION_REQUIRED;
- canSignDocument — разрешено OWNER для документов со статусом APPROVED, которые ещё не подписаны;
- canArchiveDocument — разрешено OWNER для документов со статусом APPROVED, REJECTED или SIGNED.
