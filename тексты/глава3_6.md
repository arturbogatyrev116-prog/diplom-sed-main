3.6. Система аудита и журналирования

3.6.1. Структура журнала аудита

Журнал аудита реализован на основе модели AuditLog в базе данных. В таблице 6 приведено описание полей модели.

Таблица 6 – Структура модели AuditLog

[ВСТАВИТЬ ТАБЛИЦУ ИЗ EXCEL — лист «3.6 Структура AuditLog»]

Для ускорения типичных запросов к журналу (фильтрация по пользователю, по объекту, сортировка по времени) созданы три индекса: по полю actorId, по составному ключу (entityType, entityId) и по полю createdAt.

При удалении учётной записи пользователя поле actorId в связанных записях журнала обнуляется. Это позволяет сохранить историческую информацию о действиях даже после деактивации или удаления учётной записи.

3.6.3. Дедупликация событий просмотра

Событие DOCUMENT_VIEWED генерируется при каждом открытии карточки документа. Без дедупликации это приводило бы к накоплению большого числа однотипных записей при многократном обновлении страницы.

Для решения этой проблемы реализован механизм дедупликации с окном 60 секунд. Перед записью нового события типа DOCUMENT_VIEWED функция logAuditEvent() выполняет запрос к базе данных: если за последние 60 секунд уже существует запись с тем же action, actorId и entityId, новая запись не создаётся. Дедупликация применяется только к событиям из множества DEDUP_ACTIONS (в текущей реализации — только DOCUMENT_VIEWED); все остальные события всегда записываются без проверки.

3.6.4. Отказоустойчивость

Функция logAuditEvent() обёрнута в блок try/catch. При возникновении любой ошибки (недоступность базы данных, нарушение ограничений целостности и др.) исключение перехватывается, сообщение об ошибке выводится в консоль сервера, но выполнение основного бизнес-процесса не прерывается. Такой подход обеспечивает отказоустойчивость: сбой подсистемы аудита не влияет на работоспособность системы документооборота в целом.

Вызовы logAuditEvent() в обработчиках аутентификации выполняются асинхронно через void (fire-and-forget), что исключает задержку ответа пользователю из-за операций записи в журнал.
