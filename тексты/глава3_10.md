3.10.2. Порядок контроля безопасности

Журнал аудита является основным инструментом контроля. Администратор системы имеет доступ к странице журнала аудита, где отображаются все события с фильтрацией по пользователю, типу события и временному диапазону. Особое внимание следует уделять событиям с суффиксом _DENIED — они сигнализируют о попытках несанкционированного доступа.

Верификация подписей выполняется автоматически при каждом открытии карточки документа. Результат верификации отображается непосредственно на странице документа: при успешной верификации указываются имя подписанта, его email и время подписания; при неуспешной — причина ошибки. Это позволяет немедленно выявить факт подмены содержимого документа.

Ролевые ограничения контролируются на уровне серверных политик доступа. Попытки обхода ограничений через прямые запросы к API фиксируются в журнале аудита как события с суффиксом _DENIED.

3.10.3. Выявленные ограничения и направления развития

В ходе реализации выявлен ряд ограничений, которые могут быть устранены в последующих версиях системы.

In-memory rate limiting. Счётчики ограничения частоты запросов хранятся в оперативной памяти процесса. При перезапуске сервера счётчики сбрасываются; в многопроцессорной среде счётчики не синхронизируются между экземплярами приложения. Для устранения следует использовать внешнее хранилище (Redis).

Самоподписанные сертификаты. Сертификаты ЭЦП являются самоподписанными и не верифицируются доверенным удостоверяющим центром. Для юридически значимого документооборота необходима интеграция с аккредитованным УЦ и использование квалифицированной электронной подписи (КЭП) в соответствии с Федеральным законом № 63-ФЗ «Об электронной подписи».

Хранение приватных ключей. Приватные ключи хранятся в файловой системе сервера в незашифрованном PEM-формате. Для повышения уровня защиты рекомендуется использование аппаратных модулей безопасности (HSM) или зашифрованного хранилища ключей.

Опциональность МФА. Многофакторная аутентификация является необязательной. Для повышения уровня безопасности целесообразно сделать МФА обязательной для пользователей с привилегированными ролями (OWNER, ADMIN).

SQLite как СУБД. Использование SQLite обусловлено учебным характером проекта. Для промышленного применения следует перейти на PostgreSQL или иную клиент-серверную СУБД с поддержкой конкурентного доступа и репликации.

Несмотря на перечисленные ограничения, реализованная система обеспечивает многоуровневую защиту, соответствующую базовым требованиям информационной безопасности для корпоративных систем документооборота, и может служить основой для дальнейшего развития в направлении промышленного применения.
