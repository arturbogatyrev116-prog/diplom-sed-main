## Test Plan — дипломный прототип СЭД

### Общее

- **Цель**: проверить ключевые функции и защитные механизмы прототипа (auth/MFA/documents/policy/audit/hardening).
- **Формат**: сценарии с идентификаторами, шагами и ожидаемыми результатами.
- **Среда**:
  - Для проверки **Secure Headers / CSP** использовать **production-режим**:
    - `npm run build`
    - `npm run start`
    - затем проверка через DevTools → Network → Response Headers
- **Ограничения MVP**:
  - Rate limiting: **in-memory**, **single-instance**, счётчики **сбрасываются** после перезапуска процесса.
- **Аудит**:
  - Для success/denied событий ожидаемый результат формулируется как **«событие появляется в журнале»**, без требования «ровно один раз».

## AUTH — аутентификация и сессии

### AUTH-01 — Успешный login (без MFA)
- **Что проверяется**: вход по credentials.
- **Предусловия**: пользователь активен, `mfaEnabled = false`.
- **Шаги**:
  - Открыть `/login`
  - Ввести email/пароль, отправить форму
- **Ожидаемый результат**:
  - Переход на `/dashboard`
  - Сессия активна, protected pages доступны
  - В `/audit` появляется событие **`LOGIN_SUCCESS`**

### AUTH-02 — Logout
- **Что проверяется**: выход из системы.
- **Предусловия**: пользователь залогинен.
- **Шаги**:
  - Нажать «Выйти» в topbar
- **Ожидаемый результат**:
  - Редирект на `/login`
  - Protected pages недоступны без входа
  - В `/audit` появляется событие **`LOGOUT_SUCCESS`**

### AUTH-03 — Неверный пароль
- **Что проверяется**: отказ при неверных учетных данных.
- **Предусловия**: пользователь существует.
- **Шаги**:
  - Открыть `/login`
  - Ввести корректный email + неверный пароль
- **Ожидаемый результат**:
  - Вход не происходит, показывается безопасная ошибка
  - **Нет** события `LOGIN_SUCCESS` в `/audit`

### AUTH-04 — Login с MFA (MFA_REQUIRED → ввод TOTP → успех)
- **Что проверяется**: end-to-end login при `mfaEnabled=true`.
- **Предусловия**: у пользователя включена MFA и есть секрет.
- **Шаги**:
  - Открыть `/login`
  - Ввести email/пароль → получить запрос MFA
  - Ввести корректный TOTP код
- **Ожидаемый результат**:
  - Переход на `/dashboard`
  - В `/audit` появляется событие **`LOGIN_SUCCESS`**

### AUTH-05 — Неверный TOTP при включенной MFA
- **Что проверяется**: отказ при неверном коде.
- **Предусловия**: у пользователя включена MFA.
- **Шаги**:
  - Открыть `/login`
  - Ввести email/пароль → запрос MFA
  - Ввести неверный TOTP
- **Ожидаемый результат**:
  - Вход не происходит, показывается безопасная ошибка
  - **Нет** события `LOGIN_SUCCESS` в `/audit`

## MFA — управление MFA

### MFA-01 — Включение MFA
- **Что проверяется**: настройка MFA в `/settings/mfa`.
- **Предусловия**: пользователь залогинен, `mfaEnabled=false`.
- **Шаги**:
  - Открыть `/settings/mfa`
  - Сгенерировать секрет
  - Подтвердить валидным кодом
- **Ожидаемый результат**:
  - MFA включена
  - В `/audit` появляется событие **`MFA_ENABLED`**

## DOC — документы

### DOC-01 — Список документов
- **Что проверяется**: список только своих документов.
- **Предусловия**: пользователь залогинен.
- **Шаги**:
  - Открыть `/documents`
- **Ожидаемый результат**:
  - Отображаются только документы текущего пользователя

### DOC-02 — Создание документа
- **Что проверяется**: создание + версия 1.
- **Предусловия**: роль не AUDITOR, пользователь залогинен.
- **Шаги**:
  - Открыть `/documents/new`
  - Заполнить title/type/content
  - Отправить форму
- **Ожидаемый результат**:
  - Редирект на `/documents/[id]`
  - `status = DRAFT`, `currentVersion = 1`
  - В истории есть версия 1
  - В `/audit` появляется **`DOCUMENT_CREATED`**

### DOC-03 — Просмотр документа
- **Что проверяется**: просмотр карточки документа.
- **Предусловия**: существует документ текущего пользователя.
- **Шаги**:
  - Открыть `/documents/[id]`
- **Ожидаемый результат**:
  - Данные документа видны
  - В `/audit` появляется **`DOCUMENT_VIEWED`**

### DOC-04 — Редактирование draft + рост версии
- **Что проверяется**: редактирование DRAFT, создание новой версии.
- **Предусловия**: документ текущего пользователя, `status = DRAFT`.
- **Шаги**:
  - Открыть `/documents/[id]/edit`
  - Изменить title/content
  - Сохранить
- **Ожидаемый результат**:
  - Редирект на `/documents/[id]`
  - `currentVersion` увеличился
  - В версии появляется новая запись
  - В `/audit` появляется **`DOCUMENT_UPDATED`**

### DOC-05 — Запрет редактирования non-DRAFT
- **Что проверяется**: редактирование запрещено, если статус не DRAFT.
- **Предусловия**: документ текущего пользователя со статусом не DRAFT.
- **Шаги**:
  - Открыть `/documents/[id]/edit`
- **Ожидаемый результат**:
  - `notFound()` (404-safe)
  - В `/audit` появляется **`DOCUMENT_EDIT_DENIED`** или **`DOCUMENT_UPDATE_DENIED`** (в зависимости от попытки)

## ACL — разграничение доступа (policy)

### ACL-01 — Запрет просмотра чужого документа
- **Что проверяется**: `notFound()` + denied audit.
- **Предусловия**: есть документ другого пользователя (активный).
- **Шаги**:
  - Открыть `/documents/[foreignId]`
- **Ожидаемый результат**:
  - 404-safe поведение
  - В `/audit` появляется **`DOCUMENT_VIEW_DENIED`**

### ACL-02 — Запрет редактирования чужого документа
- **Шаги**:
  - Открыть `/documents/[foreignId]/edit`
  - или попытаться отправить update action (если известен id)
- **Ожидаемый результат**:
  - 404-safe на странице edit
  - При update: безопасная ошибка без деталей
  - В `/audit` появляется **`DOCUMENT_EDIT_DENIED`** / **`DOCUMENT_UPDATE_DENIED`**

### ACL-03 — AUDITOR не создаёт документ
- **Что проверяется**: RBAC.
- **Предусловия**: пользователь с ролью AUDITOR.
- **Шаги**:
  - Открыть `/documents/new`
  - Попробовать создать документ (submit)
- **Ожидаемый результат**:
  - 404-safe для `/documents/new`
  - В `/audit` появляется **`DOCUMENT_CREATE_DENIED`**

### ACL-04 — /audit недоступен для неразрешённых ролей
- **Предусловия**: EMPLOYEE/MANAGER пользователь залогинен.
- **Шаги**:
  - Открыть `/audit`
- **Ожидаемый результат**:
  - 404-safe поведение (notFound)

## AUD — проверка журнала аудита

### AUD-01 — Просмотр /audit (ADMIN/AUDITOR)
- **Предусловия**: роль ADMIN или AUDITOR.
- **Шаги**:
  - Открыть `/audit`
- **Ожидаемый результат**:
  - Видны последние события AuditLog
  - В таблице отображаются `createdAt`, `action`, `entityType`, `entityId`, `actor`

### AUD-02 — Присутствие ключевых событий
- **Что проверяется**: события появляются после действий.
- **Шаги**:
  - Выполнить DOC-02, DOC-04, DOC-03, AUTH-02, MFA-01
  - Открыть `/audit`
- **Ожидаемый результат**:
  - В журнале **появляются** `DOCUMENT_CREATED`, `DOCUMENT_UPDATED`, `DOCUMENT_VIEWED`, `LOGOUT_SUCCESS`, `MFA_ENABLED`

## SEC — hardening

### SEC-01 — Secure headers в production
- **Среда**: `npm run build` → `npm run start`.
- **Шаги**:
  - Открыть любую страницу (например `/login`)
  - DevTools → Network → выбрать документ → Response Headers
- **Ожидаемый результат**:
  - Присутствуют `X-Frame-Options`, `X-Content-Type-Options`, `Referrer-Policy`, `Permissions-Policy`, `COOP`, `CORP`

### SEC-02 — CSP не ломает UI в production
- **Среда**: `npm run build` → `npm run start`.
- **Шаги**:
  - Открыть `/login`, `/dashboard`, `/documents`, `/audit`
  - Проверить консоль на CSP-ошибки
- **Ожидаемый результат**:
  - UI работает, критичных CSP ошибок нет

### SEC-03 — Rate limiting на чувствительных POST
- **Шаги**:
  - Сгенерировать быстрый цикл запросов (20+ POST) на:
    - `/api/auth/callback/credentials` или
    - `/api/mfa/setup` или
    - `/api/mfa/confirm`
- **Ожидаемый результат**:
  - Возникает `429 Too Many Requests` с нейтральным JSON
  - Есть заголовок `Retry-After`

### SEC-04 — Safe errors
- **Шаги**:
  - Спровоцировать ошибку рендера (например, временно сломать компонент локально для проверки)
- **Ожидаемый результат**:
  - Пользователь видит нейтральный fallback без stack trace/тех. деталей
